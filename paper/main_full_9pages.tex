\begin{abstract}
Resource-constrained Internet of Things (IoT) devices experience an inherent tradeoff between post-quantum security and energy feasibility. Ring-Learning-With-Errors lattice-based authentication and Quasi-Cyclic Low-Density Parity-Check code-based hybrid encryption offer post-quantum security but incur high computational, bandwidth, and energy costs when used per-packet on Class-1 devices with only 8 KB RAM and an 8 MHz CPU.

This paper introduces \emph{SAKE-IoT} (Session Amortization for Key Establishment in IoT), a four-phase protocol where the expensive post-quantum epoch initiation is \emph{amortizedd} over an entire session of lightweight data packets. It is an extension of the lightweight authentication protocol using Ring-LWE authentication and QC-LDPC key encapsulation by Kumari et al.~\cite{kumari2022} with: (1) a two-tier epoch state machine consisting of a heavy post-quantum handshake in Tier 1 and a lightweight per-packet AEAD in Tier 2; (2)~\HKDF{} per-packet session key derivation (RFC~5869~\cite{rfc5869}); (3)~\GCM{} authenticated encryption (NIST SP~800-38D~\cite{nist38d}); and (4) epoch renewal with Epoch Bounded Forward Secrecy (EB-FS) using secure memory zeroization.

We offer formal security proofs for IND-CCA2 (Theorem~1),
Strict Replay Resistance with $\Pr=0$ (Theorem~2), and Epoch
Bounded Forward Secrecy (Theorem~3), which are validated
through both MATLAB theoretical simulation and Contiki-NG/
Cooja network emulations over the IEEE 802.15.4 protocol
stack with 6LoWPAN


Key Results: \textbf{$\approx$99.1\%} reduction in per-packet computation latency  (7.37~ms $\to$ 0.068~ms); \textbf{45.1\%} reduction in per-packet bandwidth overhead (408~bits $\to$ 224~bits); and \textbf{33.1$\times$} times reduction in clock cycles per packet. Cooja network simulation: 99.7\% end-to-end latency reduction in the data phase, 94.7\% bandwidth saving at the session level with $N{=}20$~packets, and 100\% packet delivery ratio. Security Results: $\varepsilon = 0.0037 < 0.02$
(IND-CCA2); $\Pr[\text{replay accepted}] = 0$; and 0/100 cross-epoch key recovery (EB-FS), with two verified epoch renewal cycles.
\end{abstract}

\section{Introduction}
\label{sec:intro}

The Internet of Things (IoT) has facilitated ubiquitous connectivity among smart devices deployed across homes, transportation systems, healthcare facilities, and industrial systems. However, the security landscape is threatened by two factors: (1) the impending arrival of quantum computers jeopardizes all current public-key-based systems relying on RSA and ECC algorithms; and (2) the quantum computer replacement solutions impose prohibitive per-packet costs on Class-1 constrained devices~\cite{rfc7228}.

\subsection{Problem Context}

A hybrid scheme with Ring LWE Ring Signature Authentication (LR-IoTA) and a Quantum-Code Low Density Parity-Check (QC-LDPC) code Key Encapsulation Process (KEP) was presented by Kumari et~al.~\cite{kumari2022}. Though the scheme offers quantum-secure mutual authentication, the entire key encapsulation process with SLDSPA decoding for every data packet is mandatory, consuming \textbf{7.37~ms for every packet} (Kumari et~al. Table~7: $\Delta_{\mathrm{enc}} + \Delta_{\mathrm{dec}} = 1.5298 + 5.8430$~ms).

\subsection{The Amortization Gap}

We identify the \emph{Amortization Gap}: the costly post-quantum epoch
initiation ($\approx$22.55~ms, one-time) should occur \emph{once per session}, with
low-latency AEAD handling all subsequent packets ($\approx$0.068~ms each).
This is the main source of the novelty of SAKE-IoT.

Session amortization poses a required security query: does reusing the
Master Secret (MS) in multiple packets violate IND-CCA2,
facilitate replay attacks, or weaken forward secrecy? This paper
proves all of the above in a formal manner.

\subsection{Key Contributions}

\begin{enumerate}

    \item \textbf{SAKE-IoT Protocol:}: A comprehensive four-phase protocol that extends~\cite{kumari2022} with a two-tier session architecture, strict epoch bounds ($\Tmax = 86{,}400$~s, $\Nmax = 2^{20}$), and formal Phase 4 memory zeroization.
    \item \textbf{Three Formal Security Proofs:} : IND-CCA2 (Theorem~1), Strict Replay Resistance with $\Pr = 0$ (Theorem~2), and Epoch-Bounded Forward Secrecy via Ring-LWE hardness (Theorem~3).
     \item \textbf{99.1\% Per-Packet Latency Reduction:} from 7.37~ms $\to$ 0.068~ms, as empirically measured by MATLAB \texttt{tic/toc} over 10,000 iterations.
    \item \textbf{45.1\% Per-Packet Bandwidth Reduction:} from 408~bits (QC-LDPC $CT_0$) $\to$ 224~bits (AES-GCM nonce + tag).
     \item \textbf{33.1$\times$ Clock Cycle Reduction:} from $\approx$2.45M cycles (QC-LDPC HE) $\to$ $\approx$74K cycles (AEAD).
     \item \textbf{Epoch-Bounded Forward Secrecy (EB-FS):} A novel security property formally proven here, not addressed in~\cite{kumari2022}.
\end{enumerate}

\section{Related Work}
\label{sec:related}

\subsection{Lattice-Based Authentication for IoT}

The best computational characteristics are offered by ring LWE-based schemes. Li et~al.~\cite{li2020} proposed a lattice-based PAKE scheme. Wang et~al.~\cite{wang2019} proposed a two-factor ring LWE authentication scheme. RLizard~\cite{lee2019} showed efficient key encapsulation using ring LWE, although using classical polynomial multiplication. What is more, \textbf{none of the above amortize
post-quantum costs over a session}; a full handshake is needed per data exchange.
\subsection{Code-Based Encryption for IoT}

In the research by Hu et~al.~\cite{hu2019}, the authors analyzed the performance of QC-LDPC KEMs on FPGAs but not for the IoT's low-RAM requirements. Kumari et~al.~\cite{kumari2022} is the most relevant prior work to the presented research, with the authors' primitives being reused for the data phase with amortized AEAD.

\subsection{Session Management and Amortization}

TLS 1.3 session resumption ~\cite{rfc8446} is the inspiration for our work,
though the focus is classical crypto for high-end platforms. Prior
IoT authentication schemes ~\cite{reza2021} use symmetric keys for data,
though derived from classical (ECC) handshakes, offering no
post-quantum basis. To the best of our knowledge, SAKE-IoT
is the \textbf{first scheme to amortize Ring-LWE + QC-LDPC epoch
establishment} over a lightweight
\GCM{} data session.

\subsection{Lattice and LWE Foundations}

A \emph{lattice} $\Lambda$ is defined as
$\Lambda = \{ a_1\beta_1 + \cdots + a_i\beta_i \mid a_i \in \mathbb{Z} \} \subset \mathbb{R}^n$,
where$\beta_1,\ldots,\beta_i \in \mathbb{R}^n$ are linearly independent and form a basis, and n denotes the ambient dimension. In the \emph{Search-LWE}~\cite{regev2009} problem, the task is to recover $\delta_n$ from $(R_n, T_n = R_n\delta_n + \varepsilon_n)$. In the \emph{Decisional version} of the LWE problem, the task is to distinguish LWE samples from uniform randomness. Both versions of the LWE problem are used in the proof of Theorem 3 (EB-FS).

\subsection{Ring-LWE}

Ring-LWE is defined over $R_q = \mathbb{Z}_q[u]/(u^n+1)$. Given
$R_n \in R_q^n$ and error $\varepsilon_n \leftarrow \mathcal{G}^n_\sigma$,
pairs $(R_n, T_n)$ satisfy $T_n = R_n\delta_n + \varepsilon_n
\pmod{q}$~\cite{lyubashevsky2013,regev2009}. The Ring-LWE
\emph{search} hardness underpins Theorem~3:
$\Pr[\mathrm{findSearchLWE}(\lambda) = S_{ch}] \leq \mathrm{negl}(\lambda)$
(Theorem~2, Eq.~23 of~\cite{kumari2022}).


\subsection{QC-LDPC Code-Based Construction}

The matrix of parity checks $H_{qc}$ is in $\mathbb{F}_2^{X \times Y}$, with $X = 408$,  $Y = 816$, weight of each row is  $= 6$, weight of each column is  $= 3$, and $n_0 = 4$. The matrix is formed by circulant sub-matrices as in ~\cite{kumari2022} \S10.2. The syndrome $CT_0 = [\tilde{W}_l\,|\,I] \cdot \tilde{\varepsilon}^T$ is composed of $X = \mathbf{408}$~bits. The public key is given by $pk_{ds}$ with  $(n_0-1) \times X = 1{,}224$~bits. The SLDSPA method is capable of recovering $\tilde{\varepsilon}$, as in ~\cite{gallager1962,baldi2016}.

\subsection{HKDF-SHA256 (RFC~5869)}

A two-stage key derivation process~\cite{rfc5869}: First, the Extract phase computes a pseudorandom key as $\mathrm{PRK} = \mathrm{HMAC\text{-}SHA256}(\mathrm{salt, IKM})$. Then, the Expand phase derives the output key material as $\mathrm{OKM} = \mathrm{HMAC\text{-}SHA256}(\mathrm{PRK, info},|,i)$. Generating a 32-byte key necessitates two sequential HMAC invocations (RFC5869\S2.3).

\subsection{AES-256-GCM (NIST SP 800-38D)}

An Authenticated Encryption with Associated Data (AEAD) scheme~\cite{nist38d} utilizing a 256-bit key, a 96-bit nonce, and a 128-bit GHASH authentication tag. The per-packet overhead totals $96 + 128 = \mathbf{224}$~bits. The probability of a successful tag forgery is bounded by $\leq q_D/2^{128}$ after $q_D$ decryption queries.

\section{System and Adversary Model}
\label{sec:model}

\subsection{IoT Network Topology}

\textbf{Gateway:} Resource-rich border router; single trusted entity.
\textbf{Sensing Nodes:} RFC~7228 Class-1~\cite{rfc7228} constrained
devices (8~KB RAM, 100~KB Flash, 8~MHz CPU, battery-powered).

\subsection{Formal Security Model}

We work in the Random Oracle Model (ROM), consistent with Kumari et~al.~\cite{kumari2022} \S11.2. Adversaries are PPT.

\textbf{Adversary hierarchy} (from~\cite{kumari2022} \S11.1):
\begin{itemize}
  \item $\mathcal{A}_1$: Passive observer (public parameters only).
  \item $\mathcal{A}_2$: $\mathcal{A}_1$ + node corruption + parameter modification.
  \item $\mathcal{A}_3$: $\mathcal{A}_2$ + oracle access (adaptive chosen-ciphertext).
\end{itemize}

\textbf{Security criteria}: E1~(Unforgeability), E2~(Anonymity),
E3~(Unlinkability) inherited from~\cite{kumari2022}; E4~(IND-CCA2)
and E5~(EB-FS) proven in this paper (Theorems~1 and~3).

\subsection{Epoch Lifecycle}

Epoch bounded by dual criteria: $\Tmax = 86{,}400$~s and
$\Nmax = 2^{20}$ packets. Whichever expires first triggers Phase~4
(zeroization + re-initiation). These bounds are proof-driven:
$\Nmax$ prevents nonce exhaustion; $\Tmax$ bounds MS lifetime.

\subsection{Bernstein Polynomial Reconstruction}

The dominant cost of Algorithms~2 and~4 is sparse polynomial multiplication
($R_nY_n$, $T_n\hat{\rho}$). Kumari et~al.~\cite{kumari2022} employ
\emph{Bernstein reconstruction}: two polynomials of degree~$n$ are split into
halves $(\varepsilon_l, \varepsilon_h)$, three half-products
$C_0{=}A_0B_0$, $C_1{=}A_1B_1$, $C_2{=}A_2B_2$ are computed in parallel,
then reconstructed via $R^{\prime}_0 = C_0 + u^{n/2}C_1$ and
$C = R^{\prime}_1 + u^{n/2}C_2$. This sub-quadratic recursion reduces
hardware to only \textbf{72 Slices / 72 LUTs / 0.811~ms} on Xilinx Virtex-6
(Kumari et~al.~\cite{kumari2022} Table~3), achieving the 23\% authentication
delay improvement that underpins the 22.55~ms Phase~1 cost SAKE-IoT inherits.

\subsection{Data Encapsulation Process (DEP)}

In the protocol by Kumari et~al.~\cite{kumari2022}, each data packet requires both the Key
Encapsulation Process (KEP) and a Data Encapsulation Process (DEP):
$CT_1 = \mathrm{AES}(\mathit{ssk}, m)$, where $\mathit{ssk}$ is re-derived
from $\tilde{\varepsilon}$ via SHA in MAC-mode for \emph{every packet}.
SAKE-IoT eliminates this per-packet KEP+DEP cycle entirely: the shared $\Kmaster$
persists for the epoch (stored in volatile RAM), replacing the one-time $\mathit{ssk}$
with a durable master key. Tier~2 packets use only $K_i \leftarrow
\mathrm{HKDF}(\Kmaster, \mathit{Nonce}_i)$ --- no QC-LDPC operations whatsoever.

\section{The SAKE-IoT Protocol}
\label{sec:protocol}

\subsection{Protocol Overview}

SAKE-IoT extends~\cite{kumari2022} with a four-phase two-tier architecture
(Figure~\ref{fig:protocol}):
\textbf{Tier~1 (Phase~1):} Full post-quantum epoch establishment using
Algorithms~1--6 of~\cite{kumari2022}, executed \emph{once per epoch}
($\approx$22.55~ms amortized).
\textbf{Tier~2 (Phases~2--3):} Lightweight AEAD per packet ---
$\HKDF{} + \GCM{}$ --- executed \emph{per data packet}
($\approx$0.068~ms each).
\textbf{Phase~4:} Secure epoch termination via cryptographic zeroization.

\noindent\textbf{Critical distinction from Kumari et~al.~\cite{kumari2022}:}
The original scheme derives a session key $\mathit{ssk} = \mathrm{HMAC}(\tilde{\varepsilon})$
that is used \emph{immediately} for one encryption and then discarded.
SAKE-IoT instead stores this output as the \textbf{Master Secret} $\Kmaster$
in protected volatile RAM for the \emph{entire epoch duration} ($\leq \Tmax$,
$\leq \Nmax$ packets). All subsequent Tier~2 packets derive unique
per-packet keys $K_i = \mathrm{HKDF}(\Kmaster, \mathit{Nonce}_i)$ without
re-executing any QC-LDPC or SLDSPA operations.

\textbf{v3 Role corrections}~(from protocol specification):
(1)~RECEIVER generates QC-LDPC keys $(H_{qc}, G, pk_{ds})$;
(2)~SENDER generates $\tilde{\varepsilon} \to CT_0$;
(3)~BOTH derive $\Kmaster = \mathrm{HMAC\text{-}SHA256}(\tilde{\varepsilon})$
per Kumari et~al.~\cite{kumari2022} \S8.4.

\subsection{Phase~1: Epoch Initiation}

\textbf{Step 1.1 --- LR-IoTA Authentication:}
\begin{enumerate}
  \item SENDER: $(pk_n, sk_n) \leftarrow \mathrm{RingLWE\text{-}KeyGen}(n{=}512, q{=}2^{29}{-}3, \sigma{=}43)$
  \item SENDER: $S_n \leftarrow \mathrm{SignWithKeyword}(sk_n, K, N{=}3, pk_{1..N})$ \quad[$\Delta_{\mathrm{SG}} = 13.299$~ms]
  \item GATEWAY: $\mathrm{Verify}(S_n, K, N{=}3, pk_{1..N}) \to \mathrm{accept/reject}$ \quad[$\Delta_V = 0.735$~ms]
\end{enumerate}

\textbf{Step 1.2 --- QC-LDPC Key Encapsulation:}
\begin{enumerate}
  \item RECEIVER: $(H_{qc}, G) \leftarrow \mathrm{QC\text{-}LDPC\text{-}KeyGen}()$
        [$\Delta_{\mathrm{KeyGen}} = 0.8549$~ms]; $pk_{ds} = \tilde{W}_l$
  \item SENDER: $\tilde{\varepsilon} \leftarrow \mathrm{random\ error\ vector}$
  \item SENDER: $CT_0 = [\tilde{W}_l\,|\,I] \cdot \tilde{\varepsilon}^T$ (408~bits) $\to$ transmit
        [$\Delta_{\mathrm{Enc}} = 1.5298$~ms]
  \item RECEIVER: $\tilde{\varepsilon} \leftarrow \mathrm{SLDSPA\text{-}Decode}(CT_0, H_{qc}, G)$
        [$\Delta_{\mathrm{Dec}} = 5.8430$~ms]
  \item BOTH: $\Kmaster = \mathrm{HMAC\text{-}SHA256}(\tilde{\varepsilon})$ \quad[256-bit Master Secret]
\end{enumerate}

\textbf{Step 1.3 --- State Initialization:}
$\Tmax = 86{,}400$~s; $\Nmax = 2^{20}$; $\mathit{Ctr}_{Tx} = 0$;
$\mathit{Ctr}_{Rx} = 0$;
$AD = \mathit{DeviceID} \| \mathit{EpochID} \| \mathit{Nonce}_i$.

\subsection{Phase~2: Amortized Transmission --- Algorithm~7}

\begin{algorithm}[!t]
\caption{SAKE-Tier2-Sender}
\label{alg:sender}
\begin{algorithmic}[1]
\Require $\Kmaster$, $\mathit{Ctr}_{Tx}$, $\Tmax$, $\Nmax$, plaintext $m$, $AD$
\If{$\mathit{CurrentTime} > \Tmax$ \textbf{or} $\mathit{Ctr}_{Tx} \geq \Nmax$}
    \State \Call{ZeroizeAndRenew}{$\Kmaster, \mathit{Ctr}_{Tx}, \mathit{Ctr}_{Rx}$}
    \State \Return \textsc{EpochExpired}
\EndIf
\State $\mathit{Ctr}_{Tx} \leftarrow \mathit{Ctr}_{Tx} + 1$ \Comment{Strictly monotonic}
\State $\mathit{Nonce}_i \leftarrow \mathit{Ctr}_{Tx}$ \Comment{64-bit counter as nonce}
\State $K_i \leftarrow \mathrm{HKDF\text{-}SHA256}(\Kmaster, \mathit{Nonce}_i)$ \Comment{RFC~5869}
\State $(CT, TAG) \leftarrow \mathrm{AES\text{-}256\text{-}GCM\text{-}Enc}(K_i, \mathit{Nonce}_i, m, AD)$ \Comment{NIST SP~800-38D}
\State \Return $(\mathit{Nonce}_i, CT, TAG)$
\end{algorithmic}
\end{algorithm}

\noindent\textbf{Nonce format (NIST SP 800-38D):}
The 96-bit AES-GCM nonce is structured as $[\,\underbrace{0\cdots 0}_{32\text{ bits}}\,|\,
\underbrace{\mathit{Ctr}_{Tx}}_{64\text{ bits}}\,]$, ensuring no nonce reuse:
$\mathit{Ctr}_{Tx}$ is 64-bit and $\Nmax = 2^{20} \ll 2^{64}$, so nonce recycling
within an epoch is impossible.
The counter $\mathit{Ctr}_{Tx}$ is the direct source of the 64-bit nonce component [32-bit zero-prefix $|$ 64-bit $\mathit{Ctr}_{Tx}$] (NIST SP~800-38D).

\noindent\textbf{Nonce uniqueness guarantee:} No two Tier~2 packets within a single
epoch share a $\mathit{Nonce}_i$, because $\mathit{Ctr}_{Tx}$ is strictly
monotonically increasing and is never reset until Phase~4 terminates the epoch.

\noindent\textbf{Complexity:} $O(1)$ per packet --- no post-quantum operations.
\textbf{Cost (Sender):} $\approx 0.068$~ms (HKDF~$\approx$0.021~ms + AES-GCM~$\approx$0.047~ms, empirically measured).
\textbf{Cost (Receiver):} $\approx 0.038$~ms (HKDF~$\approx$0.014~ms + AES-GCM-Dec~$\approx$0.024~ms).

\subsection{Phase~3: Authenticated Reception --- Algorithm~8}

\begin{algorithm}[!t]
\caption{SAKE-Tier2-Receiver}
\label{alg:receiver}
\begin{algorithmic}[1]
\Require $\Kmaster$, $\mathit{Ctr}_{Rx}$, received $(\mathit{Nonce}_i, CT, TAG, AD)$
\If{$\mathit{Nonce}_i \leq \mathit{Ctr}_{Rx}$} \Comment{Replay / duplicate guard}
    \State \Return $\bot$ \Comment{DROP --- no MAC check}
\EndIf
\State $K_i \leftarrow \mathrm{HKDF\text{-}SHA256}(\Kmaster, \mathit{Nonce}_i)$
\State $m \leftarrow \mathrm{AES\text{-}256\text{-}GCM\text{-}Dec}(K_i, \mathit{Nonce}_i, CT, TAG, AD)$
\Comment{MAC before decrypt}
\If{$m = \bot$}
    \State \Return $\bot$ \Comment{Tag mismatch}
\EndIf
\State $\mathit{Ctr}_{Rx} \leftarrow \mathit{Nonce}_i$ \Comment{State update ONLY after authentication}
\State \Return $m$
\end{algorithmic}
\end{algorithm}

\noindent\textbf{AES-GCM verification detail:} AES-GCM decryption first

ciphertexts, directly supporting IND-CCA2 (Theorem~1).

\EndProcedure

\section{Security Analysis}
\label{sec:security}

We analyze SAKE-IoT in the ROM, consistent with~\cite{kumari2022} \S11.2.
SAKE-IoT Phase~1 inherits all security guarantees of LR-IoTA~\cite{kumari2022}:
unforgeability, anonymity, and unlinkability. These guarantees rest on the
formal game-based proofs of Kumari et al.: \textbf{Lemma~1}
of~\cite{kumari2022} establishes that games $G_0$ (real LR-IoTA execution)
and $G_1$ (ROM simulation replacing sign queries with random oracle programming)
are computationally indistinguishable when prime $q$ exceeds the bound
$(2^{f+1} \cdot j + \varepsilon)/(2E^f)^{1/(j-1)}$. Under this indistinguishability,
Theorem~2 of~\cite{kumari2022} (Eq.~23) formally proves:
\begin{equation}
\Pr[\mathrm{findSearchLWE}_{\mathcal{A}}(\lambda) = S_{ch}] \leq \mathrm{negl}(\lambda)
\end{equation}
i.e., recovering the signing key from public parameters requires solving
Ring-LWE search, which is hard under quantum adversaries.
SAKE-IoT Theorem~3 (EB-FS) directly inherits this hardness --- recovering
$\Kmaster = \mathrm{HMAC}(\tilde{\varepsilon})$ from past epoch$~k$ transcripts
requires inverting HMAC-SHA256 and/or solving Ring-LWE to recover $\tilde{\varepsilon}$,
both classifiable as negligible-probability events under the same Theorem~2 reduction.

We prove three new properties for the SAKE session layer.

\subsection{Theorem~1: IND-CCA2 Security}

\begin{theorem}[IND-CCA2 Security of SAKE-IoT Tier~2]
\label{thm:indcca2}
The SAKE-IoT Tier~2 data phase is IND-CCA2 secure under the AES-256
pseudorandom permutation (AES-PRP) and HMAC-SHA256 pseudorandom function
(HMAC-PRF) assumptions. For adversary $\mathcal{A}_3$ with $q_D$ decryption
oracle queries:
\begin{equation}
\begin{split}
\mathrm{Adv}_{\mathrm{IND\text{-}CCA2}}(\mathcal{A}_3)
&\le
\mathrm{Adv}_{\mathrm{PRF}}(\mathrm{HMAC\text{-}SHA256}) \\
+ \mathrm{Adv}_{\mathrm{PRP}}(\mathrm{AES\text{-}256}) 
&\quad
+ \frac{N_{\max} q_D}{2^{128}}
\le \mathrm{negl}(\lambda)
\end{split}
\end{equation}
\end{theorem}

\begin{proof}
For any decryption oracle query $CT \neq CT^*$, the receiver checks
the AES-GCM authentication tag before decryption (Algorithm~\ref{alg:receiver},
line~4). Because AES-GCM's GHASH tag covers both $(CT, AD)$, any forged
or altered $CT$ causes $\bot$ output --- $\mathcal{A}_3$ receives zero
information from all oracle queries. Stripped of useful decryption
feedback, $\mathcal{A}_3$ must guess challenge bit $b$ from $CT^*$ alone.
Since $K_i = \HKDF{}(\Kmaster, \mathit{Nonce}_i)$ is computationally
indistinguishable from uniform random (HKDF-PRF security,
RFC~5869~\cite{rfc5869}), $CT^*$ is computationally indistinguishable
from random noise. Therefore $\mathcal{A}_3$'s advantage $\varepsilon
\leq \mathrm{Adv}_{\mathrm{PRF}} + \mathrm{Adv}_{\mathrm{PRP}} +
(\Nmax \cdot q_D)/2^{128} \leq \mathrm{negl}(\lambda)$. \qed
\end{proof}

\noindent\textbf{MATLAB disclaimer~\cite{cryptoreview}:}
\emph{``The MATLAB simulation validates the architectural MAC-before-decrypt
property using real \texttt{javax.crypto.Mac} HMAC-SHA256. The formal IND-CCA2
bound is established through standard reduction to AES-PRP and HKDF-PRF
hardness~\cite{rfc5869}, not derived from the MATLAB simulation.''}

\subsection{Theorem~2: Strict Replay Resistance}

\begin{theorem}[Strict Replay Resistance]
\label{thm:replay}
For any adversary $\mathcal{A}$ re-transmitting a captured packet
$(\mathit{Nonce}_r, CT, TAG)$ where $\mathit{Nonce}_r \leq \mathit{Ctr}_{Rx}$:
\begin{equation}
\Pr[\text{Receiver accepts replay}] = 0
\end{equation}
This is a \textbf{deterministic} rejection --- not probabilistic.
\end{theorem}

\begin{proof}
Algorithm~\ref{alg:receiver} line~1 enforces the strict monotonic check
$\mathit{Nonce}_i > \mathit{Ctr}_{Rx}$. Packets with
$\mathit{Nonce}_r \leq \mathit{Ctr}_{Rx}$ are discarded before any MAC
computation or decryption. Because counters are strictly monotonically
increasing, every replayed, duplicated, or delayed packet has
$\mathit{Nonce}_r \leq \mathit{Ctr}_{Rx}$ and is unconditionally dropped.
\qed
\end{proof}

\noindent\textbf{Desynchronization (Corollary):} If packets 3--4 are lost
($\mathit{Ctr}_{Tx}=5$, $\mathit{Ctr}_{Rx}=2$), packet $\mathit{Nonce}=5$
is accepted ($5 > 2$), advancing $\mathit{Ctr}_{Rx}$ to~5. Replays of
3 or~4 are rejected ($3 \leq 5$). Counter self-heals with no state corruption.

\subsection{Theorem~3: Epoch-Bounded Forward Secrecy}

\begin{theorem}[Epoch-Bounded Forward Secrecy, EB-FS]
\label{thm:ebfs}
Let $\Kmaster^{(k)} = \mathrm{HMAC\text{-}SHA256}(\tilde{\varepsilon}_k)$
and $\Kmaster^{(k+1)} = \mathrm{HMAC\text{-}SHA256}(\tilde{\varepsilon}_{k+1})$,
where $\tilde{\varepsilon}_k$ and $\tilde{\varepsilon}_{k+1}$ are independently
sampled Ring-LWE error vectors.
\begin{equation}
\Pr[\mathcal{A}(\Kmaster^{(k+1)}) \text{ recovers any } K_i^{(k)}] \leq \mathrm{negl}(\lambda)
\end{equation}
\begin{equation}
\Pr[\mathcal{A}(\Kmaster^{(k)}) \text{ predicts any } K_j^{(k+1)}] \leq \mathrm{negl}(\lambda)
\end{equation}
\end{theorem}

\begin{proof}
\textit{Part A --- Zeroization:} At epoch end,
$\Kmaster^{(k)} \leftarrow \mathbf{0}^{32}$ (Phase~4). An adversary
arriving in Epoch~$k{+}1$ finds an already-overwritten $\Kmaster^{(k)}$.

\textit{Part B --- Ring-LWE Reduction:} Recovery of Epoch~$k$ keys
requires one of two paths: (Path~1)~Invert HMAC-SHA256 to find
$\tilde{\varepsilon}_k$ from $\Kmaster^{(k)}$ --- requires breaking
HMAC as a one-way function, negligible. (Path~2)~Re-derive
$\tilde{\varepsilon}_k$ from public parameters $R_n, T_n$ ---
equivalent to Ring-LWE search:
$\Pr[\mathrm{findSearchLWE}(\lambda) = S_{ch}] \leq \mathrm{negl}(\lambda)$
(Theorem~2, Eq.~23 of~\cite{kumari2022}). Both paths infeasible. \qed
\end{proof}

\noindent\textbf{Implementation note:}
\emph{``Hardware deployment requires \texttt{memset\_s()} (NIST SP~800-88~\cite{nist80088}
/ C11 Annex~K) to prevent physical RAM remnant recovery.''}

\noindent\textbf{EB-FS is a new security property in this paper.
Kumari et~al.~\cite{kumari2022} provide no form of forward secrecy.}

\subsection{Contiki-NG/Cooja Network Simulation}

To validate SAKE-IoT beyond theoretical simulation, we implement and

\noindent\textbf{Implementation note on Tier-2 cipher:} The Contiki-NG
network implementation employs \textbf{AES-256-GCM} (NIST SP~800-38D), 
identically matching the MATLAB theoretical model. This confirms the 
architectural footprint: a \textbf{28~byte} per-packet overhead (12-byte 
nonce + 16-byte GHASH tag), yielding the exact \textbf{45.1\%} over-the-wire
bandwidth reduction predicted. The MAC-before-decrypt order is strictly
enforced to guarantee IND-CCA2 security on the IoT device.

\subsubsection*{Metric N1 --- Authentication E2E Delay}

The Phase~1 authentication payload ($\approx$10.3~KB: Ring-LWE
polynomial arrays at $n{=}512$, ring signature components, and
QC-LDPC syndrome $CT_0$) is fragmented into \textbf{162~IEEE~802.15.4
frames} with ACK-based reliable delivery:
\begin{equation}
\Delta_{\text{auth}} = 8{,}242.68~\text{ms}
\label{eq:authdelay}
\end{equation}
This is dominated by 6LoWPAN fragmentation overhead, not computation,
confirming that session amortization is indispensable from the
\emph{network} perspective as well as the computational perspective.

\subsubsection*{Metric N2 --- Data Phase E2E Latency and Reduction}

Each Tier~2 data packet (52~bytes~UDP) traverses the network in a
single IEEE~802.15.4 frame:
\begin{equation}
\Delta_{\text{data}} = 25.096~\text{ms}
\label{eq:datadelay}
\end{equation}
\begin{equation}
\text{Network-layer E2E Reduction} =
\frac{8{,}242.68 - 25.096}{8{,}242.68} = \mathbf{99.7\%}
\label{eq:e2ered}
\end{equation}
The 99.7\% network-layer reduction corroborates and exceeds the
MATLAB computation-only figure of 99.1\% (Metric~M1), because the
data packet's single-frame transmission costs far less than the
162-fragment authentication payload.

\subsubsection*{Metric N4 --- Per-Packet Data Overhead Confirmed}

Each Tier~2 UDP frame: \texttt{type(1B) + SID(8B) + ctr(4B) + len(2B)
+ $|m|$B + GCM-tag(16B)} $= (31{+}|m|)$~bytes. For $|m|{=}13$~B:
44~B total. AEAD overhead = \textbf{28~bytes} (12-byte nonce + 16-byte tag)
--- exactly matching the MATLAB M2 calculation of 224~bits,
confirmed from Cooja packet logs.

\subsubsection*{Metric N6 --- Amortization Efficiency Ratio (AER)}

The AER measures session-level bandwidth saving over unamortized baseline:
\begin{equation}
\begin{aligned}
\text{AER}(N) &= \frac{(N-1)\cdot B_{\text{auth}}}{N\cdot (B_{\text{auth}} + b)} \\
B_{\text{auth}} &= 10{,}317\;\text{B} \\
b &= 29\;\text{B/msg}
\end{aligned}
\label{eq:aer}
\end{equation}
At $N{=}20$: AER $= \mathbf{94.7\%}$. Unamortized:
$20 \times 10{,}317 = 206{,}340$~B; SAKE-IoT amortized:
$10{,}317 + 20 \times 29 = 10{,}897$~B.
Packet Delivery Ratio: \textbf{100\%} (all packets delivered in both cycles).

\subsubsection*{Hardware Scope Note}

Cooja~Motes execute on a JVM, so reported E2E latencies represent pure
network transit times (6LoWPAN fragmentation, CSMA/CA backoff, RPL
routing). On real MSP430 hardware at 8~MHz, AES-256-GCM adds
$\approx$0.99~ms per packet --- less than 4\% of the 25~ms network
latency, thus negligible.

\noindent\textbf{Hardware Scope Limitation:}
Ring-LWE polynomial arrays at $n{=}512$ require $\approx$30.9~KB~RAM,
exceeding the 8~KB limit of RFC~7228 Class-1 devices~\cite{rfc7228}.
This is a known limitation of the base protocol~\cite{kumari2022},
not introduced by SAKE-IoT --- the identical Ring-LWE parameters are
used in Phase~1 of both systems. SAKE-IoT Tier~2 AEAD
(HKDF + AES-256-GCM) requires $<$1~KB RAM and is Class-1 compatible.
Mitigation paths for full Phase~1 deployment:
(1)~RFC~7228 \textbf{Class-2 devices} ($\geq$50~KB RAM,
$\geq$250~MHz ARM Cortex-M4) which support the full Ring-LWE
parameters at $n{=}512$;
(2)~Reduced lattice dimension $n{=}32$ (research prototype --- not
full-strength post-quantum).

\noindent\textbf{PDR scope:} The 100\% Packet Delivery Ratio is a
Cooja simulation result on a virtual lossless radio channel with
ACK-based reliable delivery. Real-world IEEE~802.15.4 environments
typically achieve 85--98\% PDR depending on link quality and
interference conditions. SAKE-IoT's Tier~2 protocol
(monotonic-counter nonce + AES-GCM MAC verification) operates
correctly at any PDR level --- lost packets are simply skipped in
the monotonic counter, self-healing without state corruption
(Corollary to Theorem~2, \S\ref{thm:replay}).

\section{Conclusion}
\label{sec:conclusion}

This paper presented SAKE-IoT, a four-phase session amortization protocol
that bridges the efficiency gap between post-quantum cryptographic strength
and IoT operational feasibility. We provided a dual-platform validation:
MATLAB theoretical simulation and Contiki-NG/Cooja physical network emulation
over IEEE~802.15.4 with 6LoWPAN/RPL. Cooja results confirm a 99.7\%
network-layer E2E latency reduction, 100\% packet delivery ratio, and 94.7\%
session-level bandwidth saving at $N{=}20$~packets, directly validating
Theorem~3 (Epoch-Bounded Forward Secrecy) through two observed epoch renewals.
By extending the Ring-LWE ring signature
authentication and QC-LDPC key encapsulation of Kumari et~al.~\cite{kumari2022}
with a two-tier epoch architecture, SAKE-IoT achieves: \textbf{99.1\%}
per-packet latency reduction; \textbf{45.1\%} per-packet bandwidth saving;
\textbf{33.1$\times$} clock cycle reduction; and three formally proven
security properties (IND-CCA2, Replay Pr=0, EB-FS). All existing security
guarantees of~\cite{kumari2022} are preserved.


\noindent\textbf{Future Work:}
(1)~\textit{FPGA synthesis:} Synthesize HKDF + AES-GCM on Xilinx Virtex-6
alongside existing SLDSPA hardware (Table~8 of~\cite{kumari2022}).
(2)~\textit{Sub-epoch rotation:} Explore the EB-FS to full PFS design
space by progressively reducing $\Nmax$ and $\Tmax$.
(3)~\textit{CRYSTALS-Kyber integration:} Replace QC-LDPC KEP with
MLWE/Kyber (NIST FIPS~203,~2024~\cite{fips203}) in Phase~1.
(4)~\textit{OSCORE/CoAP:} Integrate $\Kmaster$ epoch lifecycle with
OSCORE (RFC~8613~\cite{rfc8613}) for end-to-end PQ-secure IoT.
